## 2026-04-06
- The Makefile now provides `docker-build-amd64`, which runs `docker buildx build --platform linux/amd64 ... --load` and tags the local image as `mrsmith:amd64` for pre-production hosts that cannot run arm64 images.
- Pre-production compose now lives in [docker-compose.preprod.yaml](docker-compose.preprod.yaml) and expects a locally imported/re-tagged image `mrsmith:preprod` plus runtime env vars in `.env.preprod`; `.env.preprod.example` is the template and intentionally omits `BUDGET_APP_URL` because portal and budget share one backend origin.
- Pre-production and production now use a single backend-served static root for multiple SPAs: the portal stays at `/`, the budget app is built with Vite `base: /apps/budget/`, copied to `/static/apps/budget`, and deep links are served by the generic SPA fallback handler in [backend/internal/platform/staticspa/handler.go](backend/internal/platform/staticspa/handler.go).
- The launcher catalog’s default Budget href is now the internal path `/apps/budget/`; `BUDGET_APP_URL` remains only as an optional split-server local-development override and is no longer part of the Kubernetes sample runtime config.
- The production Docker frontend stage in [deploy/Dockerfile](deploy/Dockerfile) must copy the repo-root [tsconfig.base.json](tsconfig.base.json); without it, `pnpm -r build` inside the image fails in `packages/api-client` with `TS5083: Cannot read file '/app/tsconfig.base.json'`.
- Shared `UserMenu` component added to `@mrsmith/ui` at [packages/ui/src/components/UserMenu/UserMenu.tsx](packages/ui/src/components/UserMenu/UserMenu.tsx); shows Mr. Smith SVG avatar (inline) + "Agent {userName}" with a click-to-toggle dropdown containing Logout. Used by both portal Header and AppShell (budget and future mini-apps).
- `AppShell` now accepts `onLogout` prop and delegates user display to `UserMenu`; the old inline initials avatar and `.userArea`/`.userName`/`.avatar` CSS were removed from AppShell.
- Portal gained `@mrsmith/ui` as a workspace dependency; its Header now uses the shared `UserMenu` instead of its own avatar/logout markup. The standalone `apps/portal/public/mr-smith-avatar.svg` was deleted.
- Type-checking must use `pnpm --filter <app> exec tsc --noEmit`, never bare `npx tsc`; the global TS is 4.9.5 but workspaces require TS 5.x (`moduleResolution: "bundler"`, `allowImportingTsExtensions`). This rule is also documented in [CLAUDE.md](CLAUDE.md).
- Auth enablement now uses a backend-owned app catalog in [backend/internal/platform/applaunch/catalog.go](backend/internal/platform/applaunch/catalog.go); the first enforced launcher role is `app_budget_access`, which drives both portal visibility and budget API authorization.
- Portal API `GET /portal/apps` is now role-filtered from Keycloak claims instead of returning an empty placeholder, and the portal frontend now bootstraps Keycloak from `/config`, fetches `/api/portal/me` plus `/api/portal/apps`, and renders explicit loading/error/empty entitlement states.
- Budget API routes are now wrapped with `acl.RequireRole("app_budget_access")`, so hiding the Budget card in the portal is matched by backend enforcement.
- Both portal and budget frontends now fail closed on missing `/config` auth bootstrap values instead of silently running without browser auth.
- Kubernetes sample config now includes `KEYCLOAK_FRONTEND_URL`, `KEYCLOAK_FRONTEND_REALM`, `KEYCLOAK_FRONTEND_CLIENT_ID`, and single-origin `CORS_ORIGINS` for the backend-served portal+budget deployment.
- The production Docker build now copies portal assets to `/static` and budget assets to `/static/apps/budget`, matching the single-backend static hosting layout.
- `@mrsmith/auth-client` now owns a stateful auth snapshot instead of exposing `keycloak.token` directly from render. It publishes `status` (`loading` / `authenticated` / `reauthenticating` / `unauthenticated`) plus async `getAccessToken(minValidity?)`, and it resyncs React state on Keycloak init, refresh success, logout, and token-expiry events.
- Shared API clients should no longer capture a raw token string in React state. `@mrsmith/api-client` now accepts async `getToken` and an optional `onUnauthorized` callback; portal and budget use `getAccessToken` so idle-session refreshes update outgoing bearer tokens at request time.
- Idle-session auth failures are now handled primarily in `@mrsmith/auth-client`: budget disables React Query retries for `401`/`403`, portal shows a restoring-session state, and budget shows a temporary session-recovery shell instead of silently rendering empty data states after background `401`s.
- Do not force `login()` from every request-level `401` handler. That caused an application loop in practice. The stable frontend fix is the stateful auth snapshot plus lazy token refresh only when the token is near expiry; request-level `401`s may still surface to the screen, while background/token-expiry reauthentication remains in `@mrsmith/auth-client`.
- The backend Arak service-to-service client in [backend/internal/platform/arak/client.go](backend/internal/platform/arak/client.go) now treats upstream `401` as a technical-channel problem, not a user-session problem: it invalidates the cached client-credentials token, reacquires a fresh token, and retries the upstream request automatically up to 2 times before returning the final upstream response. Retry safety now includes buffering the outbound request body so `POST`/`PUT` proxy calls can be replayed.
- Budget proxy handlers now distinguish local user auth from upstream technical auth. In [backend/internal/budget/handler.go](backend/internal/budget/handler.go), `401/403` generated by Arak after the backend has already authenticated/authorized the user are translated to `502 Bad Gateway` with JSON `{ "error": "upstream authorization failed", "code": "UPSTREAM_AUTH_FAILED" }` instead of being passed through as if the user session were invalid.
- Budget frontend now recognizes `502` + `code=UPSTREAM_AUTH_FAILED` via [apps/budget/src/api/errors.ts](apps/budget/src/api/errors.ts) and renders the main read-only views as temporary service-unavailable states instead of empty states or user-session failures. This is wired into Home, Groups, Cost Centers, Budget list, and Budget detail pages.

## 2026-04-07
- Added [docs/DEV-GATEWAY-IMPLEMENTATION-PLAN.md](docs/DEV-GATEWAY-IMPLEMENTATION-PLAN.md), a decision-complete future implementation plan for an LLM agent to move local mini-app development to a single browser origin at `localhost:8080` with backend-owned dev proxying, canonical `/apps/<app_id>/` launcher hrefs, and static dist fallback when an app dev server is not running.
- Future dev-experience work should prefer a checked-in backend route manifest for app dev servers over adding one env var per app URL, and the intended default workflow is `pnpm dev` = backend + portal, with mini-app Vite servers started explicitly only when that app is being edited.
- `docs/mistra_db.json` is now accompanied by one self-contained per-schema dump file in `docs/` (`mistra_public.json`, `mistra_customers.json`, `mistra_migrations.json`, `mistra_templates.json`, `mistra_orders.json`, `mistra_loader.json`, `mistra_products.json`, `mistra_common.json`, `mistra_quotes.json`, `mistra_users.json`, `mistra_accounting.json`, `mistra_cart.json`); each generated file preserves the top-level `name` and `searchPath` metadata and wraps exactly one original schema entry.
- Treat `docs/mistra_db.json` as the aggregate source artifact and regenerate the per-schema files from it when the dump changes instead of hand-editing the split files independently.
- Docker Go builder images in [deploy/Dockerfile](deploy/Dockerfile) and [deploy/Dockerfile.dev](deploy/Dockerfile.dev) must track the backend toolchain from [backend/go.mod](backend/go.mod). The repo now requires Go 1.25.0, so both Dockerfiles use `golang:1.25`; leaving them at 1.23 breaks `make deploy-prod` during `go mod download`.
- Production deploy automation now lives in [scripts/deploy/prod.sh](scripts/deploy/prod.sh): it can `build`, `package`, `deploy`, and `rollback` the `linux/amd64` image used by production, and `make deploy-prod` performs local build/export plus SSH/SCP upload, remote `docker image load`, and `docker compose ... up -d --force-recreate --no-deps mrsmith`.
- Production release tarballs are named `mrsmith_amd64_<UTC timestamp>.tar`, exported locally under `artifacts/releases/`, copied to `/root/` on the server by default, and the remote host keeps only the latest `DEPLOY_RETENTION` tar files after a successful deploy.
- Production deploy configuration is intentionally local-only via `.env.deploy.prod` (template: [.env.deploy.prod.example](.env.deploy.prod.example)); the default remote compose target is `/DATI/mrsmith/docker-compose.yaml` with service `mrsmith`, and rollback uses `make rollback-prod RELEASE_TS=...` to reload an already uploaded tar.
- Compliance plan review heuristic: when a mini-app sits behind the shared auth/ACL middleware, do not force a plan to normalize middleware-generated `401`/`403` responses to JSON unless there is an intentional cross-app auth middleware change. Prefer narrowing the documented contract so plain-text middleware auth failures remain acceptable, while app-owned handler errors keep explicit JSON shapes.
- Compliance plan review heuristic: for the first DB-backed mini-app in a repo, `DSN + manual migration + test DB or mock` is not an executable plan. Approval should require a concrete dev/preprod/test database story, explicit migration application steps, and a decided handler-test strategy.
- Compliance frontend planning heuristic: when a hook can fetch both active-only and include-inactive lookup data, the React Query key must include that parameter, and edit flows for historical records must explicitly support currently referenced inactive values instead of assuming create-form dropdown behavior.
- Backend logging now uses a shared JSON `log/slog` setup in [backend/internal/platform/logging/logging.go](backend/internal/platform/logging/logging.go) with env-driven `LOG_LEVEL` (`debug|info|warn|error`, default `info`); `main` sets it as the process default logger.
- API middleware now adds request IDs to context, emits one structured access log per request, and wraps the API stack in panic recovery that logs a stack trace and returns JSON `500 {"error":"internal_server_error"}`. The current chain order in [backend/cmd/server/main.go](backend/cmd/server/main.go) is `Recover -> RequestID -> CORS -> AccessLog -> auth`.
- Request-scoped backend logs should use `logging.FromContext(r.Context())` so they inherit `request_id`, method, and path from middleware. Auth failures are now logged at `warn` without logging bearer tokens.
- Compliance handlers must not return raw PostgreSQL or driver errors to clients. Use the helper methods in [backend/internal/compliance/handler.go](backend/internal/compliance/handler.go) so DB failures are logged server-side and return `{"error":"internal_server_error"}`, while `sql.ErrNoRows` still maps to `404 {"error":"not_found"}`.
- Compliance list handlers now require an explicit `rows.Err()` check after iteration; transaction rollbacks should log only unexpected rollback failures (ignore `sql.ErrTxDone`).
- The implementation-planning reference in [docs/IMPLEMENTATION-PLANNING.md](docs/IMPLEMENTATION-PLANNING.md) now treats observability as a plan-level concern: backend plans should explicitly decide structured logging, request correlation, panic/failure logging, and sanitized 5xx behavior, and plan reviews should verify those diagnostics alongside status-code behavior.
